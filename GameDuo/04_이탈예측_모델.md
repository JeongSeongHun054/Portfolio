# 이탈(D14 리텐션) 예측 모델

## 문제 정의

**목표**: 신규 유저가 설치 후 처음 3일(D0~D3) 동안 보인 행동만으로, 그 유저가 D7~D14 사이에 다시 게임을 켤지(=리텐션 유지) 여부를 예측한다.

**왜 "처음 3일 행동"만 쓰는가 (미래 정보 누수 방지)**: 실무에서 이탈 예측 모델을 만드는 이유는 "이 유저가 이탈할 것 같으니 지금 뭔가 조치를 하자"는 데 있다. 그러려면 예측 시점(D3) 이후에 발생하는 정보(D7 이후 행동)를 학습에 사용하면 안 된다 — 이미 일어난 미래를 아는 채로 미래를 맞히는 건 실전에서 의미가 없다. 그래서 피처는 철저히 D0~D3까지만, 라벨(정답)은 D7~D14 사이 활동 여부로 분리했다.

**대상**: 관찰 기간이 최소 14일 이상 확보된 유저 480,785명 (너무 최근에 설치해서 아직 14일이 안 지난 유저는 정답을 알 수 없으므로 제외)

**라벨 정의**: `retained_d14` = D7~D14 사이 세션이 1회라도 있으면 1, 없으면 0. 실제 분포는 1(잔존) 69.1% / 0(이탈) 30.9%.

## 피처

| 피처 | 설명 |
|---|---|
| country, platform, acquisition_channel | 유저 속성 |
| sessions_d0_3 | D0~D3 세션 수 |
| avg_session_len_sec | D0~D3 평균 세션 길이 |
| events_d0_3 | D0~D3 총 행동 이벤트 수 |
| tutorial_complete_cnt | 튜토리얼 완료 횟수 |
| gacha_pull_cnt | 가챠 뽑기 횟수 |
| stage_clear_cnt | 스테이지 클리어 횟수 |
| purchases_d0_3 | D0~D3 결제 횟수 |

## 모델 및 결과

**모델**: BigQuery ML 로지스틱 회귀(`LOGISTIC_REG`), 클래스 불균형(69:31) 보정을 위해 `auto_class_weights=TRUE` 적용. 학습:검증 = 80:20.

| 지표 | 값 |
|---|---|
| Accuracy | 51.2% |
| Precision | 73.4% |
| Recall | 46.4% |
| F1 | 56.9% |
| ROC-AUC | 0.565 |

**해석**: ROC-AUC 0.565는 무작위 추측(0.5)보다는 낫지만 강한 예측력이라 보긴 어렵다. 처음엔 아쉬운 결과로 보일 수 있는데, 원인을 피처 가중치에서 확인할 수 있었다.

## 피처 중요도 (가중치) 분석 — 왜 예측력이 제한적이었는가

```
acquisition_channel = TikTok    -0.0693   (리텐션에 가장 부정적)
acquisition_channel = Organic   +0.0587   (리텐션에 가장 긍정적)
sessions_d0_3                   +0.0101
acquisition_channel = Applovin  -0.0103
acquisition_channel = Google    -0.0100
acquisition_channel = Facebook  -0.0093
purchases_d0_3                  +0.0034
gacha_pull_cnt                  +0.0027
tutorial_complete_cnt           +0.0027
country = ETC                   +0.0021
stage_clear_cnt                 +0.0012
```

**가장 중요한 발견**: 초반 행동 피처(세션수, 이벤트수, 가챠 횟수 등)보다 **유입 채널(acquisition_channel)의 계수가 압도적으로 크다.** 이는 이 프로젝트의 데이터 생성 로직과 정확히 일치한다 — 애초에 리텐션 곡선을 설계할 때 오가닉 유저는 1.3배, TikTok 유저는 0.8배 배율을 줬기 때문에(`00_프로젝트_진행일지.md` 6단계 참고), 모델이 실제 데이터에 숨겨진 "진짜 원인"을 정확히 찾아낸 것이다. 반대로 세션수/가챠횟수 같은 초반 행동 지표의 계수가 작은 이유도 설명된다 — 실제 시뮬레이션에서 리텐션은 "얼마나 열심히 놀았는가"가 아니라 "어느 채널로 들어왔는가"와 "설치 후 며칠이 지났는가"에 의해 결정되도록 설계했기 때문에, 모델이 그 구조를 있는 그대로 반영한 것.

**면접에서 설명할 포인트**: 단순히 "AUC 0.56이 나왔다"에서 멈추지 않고, 피처 가중치를 뜯어봐서 "왜 이 정도밖에 안 나왔는지" 원인을 구조적으로 설명할 수 있어야 한다는 걸 보여주는 사례다. 실제 서비스라면 이 결과는 "우리가 가진 초반 행동 로그만으로는 이탈을 잘 못 맞히니, 마케팅 채널 품질 관리에 더 집중하거나 다른 피처(예: 소셜 기능 사용, 친구 초대 여부)를 추가로 수집해야 한다"는 액션으로 이어질 것이다.

## 비즈니스 활용 관점: 위험도 10분위 분석

AUC는 낮지만, 예측 확률로 유저를 10개 그룹(risk decile)으로 나눠보면 실질적인 타겟팅 가치는 있다.

| 위험도 그룹 | 유저수 | 실제 D14 리텐션 |
|---|---|---|
| 1분위 (가장 안전) | 48,079 | 77.9% |
| 5분위 (중간) | 48,079 | 67.5% |
| 10분위 (가장 위험) | 48,078 | 58.8% |

가장 안전한 그룹과 가장 위험한 그룹의 실제 리텐션 차이는 약 19%p — 정교하진 않아도 "위험도 상위 10% 유저에게 리텐션 캠페인(복귀 보상 등)을 우선 발송한다" 같은 실무적 액션에는 활용 가능한 수준의 변별력이다.

## 한계 및 개선 방향 (알아둘 점)

- 이 프로젝트에서는 리텐션이 "채널×경과일" 규칙으로만 생성되어서, 실제 게임처럼 "초반에 재미를 느꼈는가"가 리텐션에 영향을 주는 구조가 약하게 반영되어 있다. 실제 서비스 데이터라면 초반 행동 피처의 예측력이 이보다 훨씬 강하게 나올 가능성이 높다.
- 개선 방향으로는 트리(BOOSTED_TREE_CLASSIFIER) 모델로 비선형 상호작용을 포착하거나, 피처를 더 세분화(예: 첫 세션 완료 여부, 첫날 이탈 시점)하는 것을 고려할 수 있다.
