# 프로젝트 진행 일지 — 게임듀오 게임 데이터 분석가 포트폴리오

이 문서는 프로젝트를 처음부터 끝까지 어떤 생각으로, 어떤 순서로 만들었는지 전부 기록한다. 나중에 이력서/면접에서 "왜 이렇게 만들었나요?"라는 질문에 이 문서를 보면서 그대로 설명할 수 있도록 하는 것이 목적이다.

---

## 1단계. 지원 공고 분석 (JD 분석)

**무엇을 했나**: 게임듀오 "게임 데이터 분석가" 채용 공고 원문을 그대로 분석해서 필수 역량과 우대 역량을 정리했다. (`01_JD_분석.md` 참고)

**왜 이걸 먼저 했나**: 포트폴리오를 아무렇게나 만들면 "내가 하고 싶은 걸 보여주는 프로젝트"가 되기 쉽다. 그런데 채용 담당자가 실제로 보고 싶은 건 "이 사람이 우리 회사에서 요구하는 일을 할 수 있는가"이다. 그래서 프로젝트를 설계하기 전에 JD를 문장 단위로 뜯어서, 어떤 기술/경험이 필수이고 어떤 게 우대사항인지부터 명확히 정리했다.

**핵심 발견**:
- 필수: Python/SQL 데이터 핸들링, GCP(BigQuery)/AWS 대용량 데이터 경험, BI Tool(Looker Studio/Tableau), A/B 테스트 설계
- 우대: 게임 지표(LTV, ARPU, CPI) 이해, 인과추론, Airflow/dbt, 생성형 AI 활용

이 발견이 이후의 모든 결정(장르 선택, 데이터 규모, 클라우드 연동 여부)의 기준이 되었다.

---

## 2단계. 어떤 게임을 만들지 결정

**결정**: 실제 데이터를 구하는 대신 가상의 방치형(Idle) RPG 게임 데이터를 직접 시뮬레이션해서 만들기로 함.

**왜 시뮬레이션인가**: 공개된 게임 데이터셋은 대부분 장르가 안 맞거나 (예: PC/콘솔 게임), 지표 정의가 실무와 다르거나, 스키마가 이미 정제되어 있어서 "로그 설계 → 정제 → 분석"이라는 JD가 요구하는 전체 프로세스를 보여줄 수 없다. 반면 직접 시뮬레이션하면 로그 설계 단계부터 관여했다는 걸 증명할 수 있다.

**왜 하필 방치형 RPG인가**: 게임듀오가 실제로 만드는 게임(아처키우기, 캣히어로, 어나던던전)이 전부 방치형 RPG 계열이다. 지원 동기 문서나 면접에서 "이 회사 게임의 BM 구조를 이해하고 포트폴리오를 설계했다"고 말할 수 있는 게 일반 장르보다 훨씬 설득력 있다고 판단했다.

**게임 컨셉 상세**: `02_데이터_스키마_설계.md` 참고. 오토 전투, 가챠(뽑기), 강화, 스태미나, 일일퀘스트, 상점(IAP), 리워드 광고 등 방치형 RPG의 표준 F2P 시스템을 설계에 반영했다.

## 3단계. 데이터 스키마 설계

**결정한 테이블**: `users`(유저 마스터), `sessions`(세션 로그), `events`(행동 이벤트 로그), `purchases`(결제 로그), `marketing_spend`(마케팅 비용), `ab_test_assignment`(A/B 테스트 배정)

**왜 이렇게 나눴나**: 실제 게임사 데이터 구조를 최대한 흉내냈다. 유저 정보(변하지 않는 속성)와 행동 로그(계속 쌓이는 이벤트)를 분리하는 건 실무에서 기본적인 설계 원칙이다. 이렇게 분리해야 "유저 세그먼트별로 행동이 어떻게 다른가" 같은 분석을 JOIN으로 자연스럽게 할 수 있다.

**유저 세그먼트 설계**: 실제 F2P 게임의 전형적인 패턴을 반영해서 무과금 95% / 소액과금(미노우) 3% / 중간과금(돌핀) 1.5% / 고액과금(웨일) 0.5%로 나눴다. 이렇게 하는 이유는, 상위 소수 과금 유저가 매출 대부분을 차지하는 게 실제 모바일 게임의 특징이고, 이런 편차가 있어야 나중에 "과금 세그먼트별 LTV 분석" 같은 유의미한 분석 결과가 나온다.

## 4단계. 데이터 규모를 대규모로 키우기로 결정

**계기**: JD 자격요건에 "GCP(BigQuery), AWS 등 클라우드 환경에서 대용량 데이터 활용해본 경험"이 명시되어 있는 걸 발견하고, 원래 계획했던 소규모(5천명) 데이터로는 이 요건을 제대로 증명할 수 없다고 판단해서 규모를 키우기로 결정함.

## 5단계. 로컬 시뮬레이션이 아니라 진짜 GCP BigQuery를 쓰기로 결정

**왜 진짜 클라우드를 쓰나**: "클라우드 환경에서 대용량 데이터를 다뤄본 경험"이라는 JD 문장은 실제로 클라우드를 써본 사람이 훨씬 설득력 있게 답할 수 있다. 로컬 파이썬으로 만든 뒤 "BigQuery 문법으로 작성했다"고 하는 것보다, 실제 GCP 프로젝트를 만들고 그 안에서 데이터를 다룬 경험 자체가 면접에서 훨씬 강한 답변이 된다.

**진행 과정**: GCP 프로젝트(`newproject-502815`) 생성 → BigQuery API 활성화 → claude.ai 커넥터로 BigQuery 인증 연결 → 파티셔닝/클러스터링을 반영한 스키마 설계 (자세한 개념 설명은 아래 "배운 개념" 섹션 참고)

## 6단계. BigQuery 안에서 직접 데이터 생성

**방법론 결정**: 로컬에서 파일을 만들어 업로드하는 대신, BigQuery SQL 자체의 대량 데이터 생성 기능(`GENERATE_ARRAY`, `UNNEST` 등)을 활용해서 BigQuery 안에서 직접 수백만 건의 데이터를 만들기로 함. 파일 업로드 도구가 없었던 것도 이유지만, 오히려 이 방식이 "SQL로 대용량 데이터를 다룬다"는 스토리에 더 잘 맞기도 함.

**현실적인 패턴을 만들기 위한 두 가지 핵심 모델**:

1. **리텐션 곡선**: 설치 다음날(D1) 45% 잔존, 7일 후(D7) 20%, 그 이후는 서서히 감소하다 "꾸준히 하는 코어 유저" 6%가 오래 남는 구조로 설계. 공식: `retention(d) = max(0.06, 0.515 × exp(-0.1353 × d))`, 단 설치 당일(d=0)은 100%. 유입 채널에 따라 보정치도 적용 (오가닉 유저는 리텐션이 더 좋고, TikTok 유입은 더 낮게 설정 — 실제로 유료 광고 채널별로 유저 품질이 다르다는 업계 통념을 반영).

2. **과금 세그먼트별 결제 확률**: 무과금은 0%, 미노우는 하루 1% 확률로 소액 결제, 돌핀은 3% 확률로 중간 결제, 웨일은 8% 확률로 고액 결제.

**검증**: 실제로 생성된 데이터의 D1/D7/D30 리텐션을 계산해서 목표치(45%/20%/8%)와 비슷하게 나왔는지 확인함 (실제 결과: D1 48%, D7 21.5%, D30 6.5% — 설계 목표와 거의 일치).

## 7단계. 대용량 파티션 테이블에서 데이터가 사라지는 문제 발생 및 해결 (중요한 문제 해결 사례)

이 부분은 실제 데이터 분석/엔지니어링 업무에서 겪을 법한 "원인을 알 수 없는 데이터 손실"을 처음부터 끝까지 체계적으로 추적해서 해결한 과정이라 **면접에서 문제 해결 능력을 보여줄 수 있는 좋은 사례**가 될 수 있다. 순서대로 정리한다.

**증상**: 736만 건이 나와야 할 세션 데이터가 매번 205만 건 정도로 줄어들어서 나옴. 여러 번 다시 만들어도 계속 비슷한 숫자로 줄어듦.

**가설 1 (기각)**: "우리가 정한 가상의 '오늘 날짜'와 실제 시스템 날짜가 달라서 필터링이 잘못 걸렸다" → 날짜를 고정값으로 바꿔서 테스트했지만 결과가 똑같아서 기각.

**가설 2 (기각)**: "무작위 확률(RAND 함수)이 대용량 쿼리에서 불안정하게 동작한다" → 순수하게 결정론적인 해시 기반 난수(FARM_FINGERPRINT)로 바꿔서 테스트했지만 여전히 같은 문제 발생, 기각.

**가설 3 (기각)**: "쿼리를 여러 단계(CTE)로 나눠서 쓰면 계산이 꼬인다" → 한 단계로 합쳐서 순수 SELECT로 검증했더니 정상적으로 736만 건이 나옴. 하지만 이 로직을 그대로 파티션 테이블(`PARTITION BY`) 생성에 적용하면 다시 205만 건으로 줄어드는 걸 발견 → 즉 문제는 "파티션 테이블 생성" 자체에 있다는 걸 좁혀냄.

**결정적 단서**: 파티션 테이블에 실제 데이터를 입력(INSERT)하려고 시도했더니 "이 프로젝트에 결제 계정이 연결되지 않아서 데이터 조작(DML) 쿼리를 쓸 수 없다"는 에러가 명확하게 뜸. 이걸 보고 "지금까지의 데이터 손실도 결제 미연결로 인한 샌드박스 제한 때문일 것"이라는 가설을 세움.

**진짜 원인**: 사용자가 결제 계정을 연결한 뒤 테이블 메타데이터를 직접 조회해보니, 데이터셋 자체에 `defaultPartitionExpirationMs: 60일`이라는 설정이 걸려있는 것을 발견. 이건 결제 계정이 연결되지 않은 "BigQuery 샌드박스 모드"에서 데이터셋을 만들면 구글이 자동으로 붙이는 정책이다 — "60일이 지난 데이터는 자동으로 삭제"하는 설정. 오늘 날짜(2026-07-19) 기준 60일 전인 2026-05-20 이전 데이터가 계속 자동 삭제되고 있었던 것.

**해결**: 데이터셋 설정에서 `ALTER SCHEMA ... SET OPTIONS (default_partition_expiration_days = NULL, ...)` 명령으로 자동 삭제 정책을 제거. 이미 만들어진 테이블들도 개별적으로 만료 설정을 해제하고, 세션 테이블은 다시 생성해서 정확한 736만 2,135건을 확보함.

**이 경험에서 배운 것 (면접 답변 포인트)**: 예상과 다른 결과가 나왔을 때 바로 "왜 이런지 모르겠다"고 넘어가지 않고, 가능한 원인들을 하나씩 가설로 세워서 순서대로 배제해나가는 방식으로 접근했다. 특히 눈에 보이는 에러 메시지가 없는 상황(조용히 데이터가 줄어드는 상황)에서 작은 샘플로 실험을 쪼개서 어느 단계에서 문제가 생기는지 좁혀나간 게 핵심이었다. 실무에서도 "에러는 없는데 숫자가 이상하다"는 상황은 매우 흔하고, 이런 체계적 접근이 실제 데이터 분석가에게 요구되는 역량이다.

## 배운 핵심 개념 정리 (비전공자도 이해할 수 있게)

**파티셔닝(Partitioning)**: 거대한 테이블을 통째로 두지 않고 날짜 같은 기준으로 미리 "서랍"처럼 나눠서 저장하는 것. "최근 7일만 보여줘" 같은 쿼리를 할 때 관련 없는 서랍은 아예 열어보지 않아도 되니까 빠르고 저렴해짐.

**클러스터링(Clustering)**: 파티션(서랍) 안에서도 특정 기준(예: 유저ID)으로 미리 정렬해서 저장해두는 것. 관련된 데이터가 물리적으로 가까이 모여있어서 조회가 빨라짐.

**BigQuery 샌드박스 모드**: 결제 계정을 연결하지 않고 GCP 프로젝트를 만들면 자동으로 적용되는 무료 체험 모드. 데이터 조작 쿼리(INSERT/UPDATE/DELETE) 불가, 저장 용량 10GB 제한, 그리고 이번에 직접 겪은 것처럼 **데이터셋에 60일 자동 삭제 정책이 기본으로 걸림**. 결제 계정을 연결하면 이 제한들이 풀리지만, 이미 만들어진 데이터셋/테이블에 남아있는 기존 설정은 자동으로 사라지지 않고 직접 해제해줘야 함.

## 8단계. 나머지 테이블(events, purchases, ab_test_assignment) 생성

**events (행동 이벤트 로그)**: 세션마다 3~22개(평균 12.5개)의 행동 이벤트를 발생시켜서 총 9,202만 건을 생성했다. 이벤트 종류는 stage_clear(30%), currency_earn(20%), currency_spend(10%), item_upgrade(10%), gacha_pull(8%), quest_complete(8%), ad_watch_reward(7%), level_up(4%), shop_view(2%), tutorial_complete(1%) 비율로 배분했다 — 방치형 RPG에서 실제로 스테이지 클리어와 재화 획득이 가장 빈번한 행동이라는 점을 반영.

**purchases (결제 로그)**: 처음에는 세션당 결제 확률을 미노우 1%/돌핀 3%/웨일 8%로 낮게 잡았는데, 계산해보니 결제 세그먼트로 분류된 유저 중 상당수가 평생 단 한 번도 결제를 안 하는 모순이 생겼다(세그먼트 라벨이 "이 유저는 결제자다"라는 뜻인데 실제로는 결제를 안 하는 상황). 그래서 확률을 미노우 12%/돌핀 20%/웨일 35%로 올려서, 각 세그먼트 유저가 평생 동안 최소 한 번 이상 결제할 가능성이 현실적으로 높아지도록 재조정했다. 최종 62,261건 생성.

**ab_test_assignment (A/B 테스트 배정)**: 실제 게임사에서 실험을 상시 진행하는 게 아니라 특정 기간에 신규 유입 유저를 대상으로 진행하는 걸 반영해서, 두 개의 가상 실험을 설계했다.
- `tutorial_v2`: 2025-10-01~10-31 설치 유저 대상 (컨트롤 21,057 / 트리트먼트 21,110)
- `gacha_rate_boost`: 2026-02-01~02-28 설치 유저 대상 (컨트롤 19,162 / 트리트먼트 19,303)

두 실험 다 컨트롤/트리트먼트 비율이 50:50에 가깝게 잘 나뉜 것을 확인했다.

## 9단계. 데이터 정제/가공 — 분석용 요약 테이블 만들기

**왜 필요한가**: raw 테이블(특히 events 9,202만 건)에 매번 직접 집계 쿼리를 날리면 느리고 비용도 계속 든다. 실무에서는 자주 조회하는 지표를 미리 계산해서 별도 테이블(데이터 마트)로 만들어두고, 대시보드는 이 요약 테이블만 보고 빠르게 그리는 방식을 쓴다. 그래서 4개의 요약 테이블을 만들었다.

1. **`daily_kpi_summary`** (날짜별 핵심지표): DAU, 신규가입자, 매출, 결제자수, ARPU, ARPPU. 검증 결과 DAU 약 3.5만, ARPU $0.3~0.4/일, ARPPU $40~50 수준으로 합리적인 수치가 나옴.

2. **`user_summary`** (유저별 요약): 유저 50만 명 각각의 총 세션수, 총 이벤트수, 총 매출, 마지막 활동일, 14일 이상 미접속 시 이탈(churn) 플래그. 이 테이블은 나중에 이탈/LTV 예측 모델을 만들 때 그대로 피처(feature) 테이블로 재사용할 계획.

3. **`cohort_retention`** (설치 주차별 리텐션 곡선): 검증 결과 D1 47.3%, D7 20.7%, D30 6.3%, D60~D90 6.2~6.3%로 안정화 — 원래 설계 목표(D1 45%, D7 20%, D30 8%)와 거의 일치하는 걸 확인.

4. **`channel_performance`** (마케팅 채널 효율): 채널별 CPI, D7 리텐션, ROAS를 계산. 결과에서 의미 있는 패턴이 자연스럽게 나타남 — Organic 채널은 스펜드 없이 D7 리텐션 25.5%(가장 높음), TikTok은 D7 리텐션 15.9%(가장 낮음, 애초에 리텐션 배율을 0.8배로 설계했기 때문), Applovin은 CPI가 가장 낮으면서 ROAS는 가장 높음(1.4) — 이런 채널별 차이가 나중에 "어느 채널에 예산을 더 써야 하는가" 같은 인사이트 분석의 재료가 된다.

**알아둘 점(설계 상 단순화)**: `channel_performance`의 스펜드/설치 수는 `marketing_spend` 테이블에서, 실제 유저 수/매출은 `users`+`user_summary`에서 따로 계산해서 합친 것이다. 실제 서비스에서는 마케팅 스펜드와 실제 유저 유입이 어트리뷰션(attribution) 로직으로 1:1 연결되지만, 이 포트폴리오에서는 두 데이터를 각각 독립적으로 생성했기 때문에 완벽한 인과관계는 아니고 통계적으로 비슷한 패턴을 보이는 수준이다. 이 부분은 나중에 리포트에 "실제 환경이라면 어트리뷰션 데이터를 연결해서 더 정교하게 분석했을 것"이라고 설명할 수 있는 포인트.

## 10단계. 대시보드 제작

**만든 것**: `게임듀오_대시보드.html` — BigQuery 요약 테이블에서 집계한 데이터를 임베드한 자체완결형 인터랙티브 대시보드. 브라우저로 더블클릭만 하면 바로 열리고, 별도 서버나 로그인 없이 누구에게나 공유 가능.

**구성**: 핵심 KPI 카드 6개(누적 유저/매출/결제유저/세션/D1·D7 리텐션), 일별 DAU·매출 추이(기간 필터 지원), 코호트 리텐션 곡선, 과금 세그먼트별 매출 비중(도넛), 국가별 유저/매출, 채널별 ROAS·D7 리텐션 비교, 채널 성과 상세 테이블.

**왜 이 지표들을 선택했나**: JD에서 요구한 "지표의 흐름을 한눈에 보여주는 BI 대시보드"에 맞춰, 게임 운영에서 실제로 매일 보는 지표(DAU, 매출, 리텐션)와 의사결정에 필요한 지표(세그먼트별 매출 기여도, 채널별 ROAS)를 함께 배치했다. 특히 채널별 ROAS 차트는 "어느 채널에 마케팅 예산을 더 써야 하는가"라는 실제 비즈니스 질문에 바로 답할 수 있도록 설계했다.

**실제 Looker Studio 연동은 별도로 진행 권장**: 이 HTML 대시보드는 빠른 공유용 결과물이고, JD에 명시된 "Looker Studio 활용 경험"을 실제로 쌓으려면 BigQuery를 라이브 데이터 소스로 연결하는 것도 해볼 가치가 있다. 방법: Looker Studio(lookerstudio.google.com) → 새 보고서 만들기 → 데이터 소스 추가 → BigQuery 커넥터 선택 → 프로젝트 `newproject-502815` → 데이터셋 `dungeon_game_analytics` → 원하는 테이블(`daily_kpi_summary` 등) 선택. 이렇게 연결하면 데이터가 바뀔 때마다 대시보드가 자동으로 갱신되는 "라이브 대시보드" 경험을 실제로 해볼 수 있다.

## 11단계. A/B 테스트 분석

**고민했던 지점**: 애초에 `ab_test_assignment`는 랜덤 배정만 해뒀을 뿐, 실제 컨트롤/트리트먼트 간 행동 차이는 데이터 생성 과정에 반영되어 있지 않았다. 이 상태로 분석하면 두 실험 다 "차이 없음"이 나올 가능성이 높았다. 그래서 `gacha_rate_boost` 실험에는 트리트먼트 그룹의 무과금 유저 중 3%를 결제 전환시키는 방식으로 현실적인 효과를 의도적으로 주입했고, `tutorial_v2`는 아무 조작 없이 자연 상태 그대로 뒀다. 이렇게 하면 "유의미한 효과를 정확히 탐지하는 경우"와 "효과 없음을 정확히 판단하는 경우"를 둘 다 보여줄 수 있다.

**결과**:
- `gacha_rate_boost` (결제 전환율): 컨트롤 4.42% vs 트리트먼트 7.63%, z=13.23, **p<0.000001 → 통계적으로 매우 유의미**
- `tutorial_v2` (D1 리텐션): 컨트롤 47.01% vs 트리트먼트 46.78%, z=-0.47, **p=0.64 → 유의미하지 않음**

자세한 가설, 방법론, Action Plan은 `03_AB테스트_분석.md`에 별도로 정리했다.

**면접 포인트**: A/B 테스트에서 "유의미한 차이 없음"이라는 결과도 정당한 결론이라는 것, 그리고 표본이 커도(2만 명 이상) 실제 효과가 없으면 검정에서 정확히 걸러낸다는 걸 직접 확인해본 경험으로 설명할 수 있다.

## 12단계. 이탈(D14 리텐션) 예측 ML 모델

**설계 원칙**: 설치 후 첫 3일(D0~D3) 행동만으로 D7~D14 리텐션을 예측하도록 만들었다. 예측 시점 이후의 정보(미래)를 피처에 섞으면 실전에서 의미 없는 모델이 되기 때문에, 피처와 라벨의 시간 구분을 명확히 했다.

**모델**: BigQuery ML 로지스틱 회귀(`CREATE MODEL`), 클래스 불균형 보정(auto_class_weights).

**결과**: ROC-AUC 0.565로 예측력 자체는 강하지 않았지만, 원인을 피처 가중치 분석으로 정확히 짚어냈다 — acquisition_channel(유입 채널)의 계수가 압도적으로 컸고, 이는 애초에 리텐션 곡선을 채널별 배율로 설계했기 때문이었다(모델이 데이터에 숨겨진 진짜 원인을 정확히 재발견한 것). 위험도 10분위로 나눠보면 최상위/최하위 그룹 간 실제 리텐션 차이가 19%p로, 실무 타겟팅에는 활용 가능한 수준.

**중간에 있었던 해프닝**: 클래스 가중치 옵션을 껐다 켰다 하며 두 모델을 비교하다가, BigQuery의 쿼리 캐시 때문에 같은 텍스트의 평가 쿼리가 이전 모델의 결과를 그대로 반환하는 걸 놓칠 뻔했다. 쿼리에 주석을 하나 추가해서 캐시를 우회한 뒤에야 진짜 결과를 확인할 수 있었다 — "숫자가 이상하게 똑같다"를 그냥 넘기지 않고 검증하는 습관이 여기서도 도움이 됐다.

자세한 내용은 `04_이탈예측_모델.md`에 정리했다.

## 13단계. 자동화 파이프라인 (dbt + Airflow) — JD 우대사항

**계기**: JD 우대사항에 "Airflow, dbt 등을 활용해 데이터 파이프라인을 자동화하고 운영해보신 분"이 있어서, 지금까지 BigQuery 콘솔에서 직접 실행했던 요약 테이블 생성 SQL을 dbt 모델(staging→marts 계층)로 재구성하고, 이를 매일 실행하는 Airflow DAG를 설계했다.

**만든 것**:
- `dbt_project/`: staging 모델(원본 테이블 1:1 정제) → marts 모델(4개 요약 테이블), 데이터 품질 테스트(not_null/unique/accepted_values) 포함
- `airflow_dags/gameduo_pipeline_dag.py`: 원본 데이터 신선도 체크 → dbt run → dbt test → DAU 이상치 체크 → 대시보드 갱신, 5단계로 이어지는 DAG

**투명하게 밝힌 한계**: 이 세션 환경에는 상시 서버(Airflow 스케줄러)를 띄울 수 없고, dbt가 BigQuery에 붙으려면 서비스 계정 키가 필요한데 지금은 OAuth 커넥터로만 연결되어 있어서 실제로 `dbt run`을 실행해보지는 못했다. 이 사실을 숨기지 않고 `05_자동화_파이프라인.md`에 정직하게 적어뒀다 — 코드는 완성했지만 실제 배포는 하지 않았다는 것, 그리고 무엇이 갖춰지면 배포 가능한지까지 명시했다.
