# 데이터 파이프라인 자동화 (dbt + Airflow)

JD 우대사항 "Airflow, dbt 등을 활용해 데이터 파이프라인을 자동화하고 운영해보신 분"에 대응하기 위해 만든 산출물이다. 지금까지는 대시보드/분석용 요약 테이블(`daily_kpi_summary`, `user_summary`, `cohort_retention`, `channel_performance`)을 BigQuery 콘솔에서 SQL을 직접 실행해서 만들었는데, 이 방식은 실무에서는 잘 안 쓴다 — 매일 최신 데이터로 자동으로 갱신되어야 하고, 여러 사람이 코드를 리뷰하고 버전 관리할 수 있어야 하기 때문이다. 그래서 같은 로직을 **dbt(변환 계층)** 와 **Airflow(스케줄링/오케스트레이션)** 로 재구성했다.

## 폴더 구성

- `dbt_project/` — 원본 테이블을 정제·집계하는 SQL 변환 로직을 dbt 모델로 재작성
- `airflow_dags/gameduo_pipeline_dag.py` — 매일 새벽 파이프라인을 실행하는 Airflow DAG 정의

## dbt: staging → marts 계층 구조

- `models/staging/` : `users`, `sessions`, `events`, `purchases` 등 원본 테이블을 최소 가공해서 그대로 노출하는 뷰. 원본 스키마가 바뀌어도 이 레이어만 고치면 된다.
- `models/marts/` : 실제 대시보드/분석에 쓰는 4개 요약 테이블. 전부 `{{ ref(...) }}`로 staging을 참조하기 때문에, 어떤 테이블이 어떤 원본에서 파생됐는지 dbt가 자동으로 계보(lineage)를 그려준다.
- `models/staging/_sources.yml`, `models/marts/_schema.yml`에 `not_null`, `unique`, `accepted_values` 같은 데이터 품질 테스트를 정의해뒀다. 예를 들어 `payer_segment` 컬럼은 반드시 4개 값(`non_payer`/`minnow`/`dolphin`/`whale`) 중 하나여야 한다는 걸 코드로 강제한다 — 데이터에 오타나 이상값이 들어오면 파이프라인이 자동으로 실패하고 알려준다.

## Airflow DAG 설계

`gameduo_daily_pipeline`은 매일 새벽 3시(전날 데이터가 다 쌓인 뒤)에 아래 순서로 실행되도록 설계했다.

1. **check_source_freshness**: 어제 날짜 세션 데이터가 실제로 들어왔는지 확인 (안 들어왔으면 이후 단계를 돌 필요가 없음)
2. **dbt_run**: staging → marts 전체 재계산
3. **dbt_test**: not_null/unique 등 데이터 품질 테스트. 실패하면 여기서 파이프라인이 멈춘다.
4. **check_dau_anomaly**: 전날 DAU가 최근 7일 평균 대비 30% 이상 급락하면 실패 처리 — 실제 장애(로그 유실, 서버 이슈 등)를 조기에 감지하기 위한 운영용 체크
5. **refresh_dashboard_data**: 최신 집계 데이터를 대시보드에 반영

각 단계가 실패하면 다음 단계로 넘어가지 않도록 순서를 명시적으로 연결(`>>`)했고, 실패 시 담당자에게 이메일 알림이 가도록 설정했다.

## 솔직히 밝혀둘 점

이 DAG와 dbt 프로젝트는 **코드로는 완성되어 있지만, 실제 Airflow 서버에 배포되어 매일 자동 실행되고 있지는 않다.** 이 프로젝트 작업 환경에는 상시 실행되는 서버(Airflow 스케줄러)를 띄울 수 없고, dbt가 BigQuery에 연결하려면 GCP 서비스 계정 키 파일이 필요한데 이 세션에서는 BigQuery에 OAuth 기반 커넥터로만 연결되어 있어서 키 파일이 없다.

그래서 이 두 산출물은 "이런 구조로 자동화 파이프라인을 설계하고 코드로 작성할 수 있다"는 걸 보여주는 결과물이며, 실제로 돌려보려면 다음이 필요하다:
1. GCP 서비스 계정 키 발급 → `dbt_project/profiles.yml.example`을 참고해 `~/.dbt/profiles.yml` 구성
2. Cloud Composer(GCP의 managed Airflow) 또는 자체 Airflow 서버에 `gameduo_pipeline_dag.py` 배포

면접에서 이 부분을 질문받으면 "실제 운영 환경 없이도 파이프라인을 코드로 설계할 줄 안다"는 것과 "무엇이 갖춰지면 실제로 배포할 수 있는지" 정확히 아는 것 자체가 포인트가 될 수 있다.
