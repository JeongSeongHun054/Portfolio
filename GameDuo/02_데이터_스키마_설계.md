# 가상 게임 컨셉 & 데이터 스키마 설계

## 1. 가상 게임 컨셉: "던전키우기" (가칭, 방치형 RPG)

GameDuo의 실제 게임(아처키우기 등)과 동일한 장르로, 방치형 RPG의 표준 F2P 시스템을 반영.

**핵심 시스템**
- 오토 전투 / 스테이지 진행 (방치 수익)
- 캐릭터 가챠(뽑기) — 확률형 아이템
- 캐릭터/장비 강화
- 스태미나(에너지) — 세션당 플레이 제한
- 일일 퀘스트 / 출석 보상
- 상점 (유료 재화 다이아 구매 = IAP)
- 리워드 광고 시청
- 시즌 이벤트

**유저 세그먼트** (데이터에 현실적 편차를 주기 위함)
- 유입 채널: Google UAC, Facebook, Applovin, TikTok, Organic — 채널별 CPI/리텐션 차등
- 국가: KR, US, JP, 기타 — 국가별 ARPU 차등
- 플랫폼: iOS / Android
- 결제 세그먼트: 무과금(Non-payer) / 미노우(Minnow) / 돌핀(Dolphin) / 웨일(Whale) — 전형적 F2P 분포 (과금유저 3~5%, 상위 0.5%가 매출 40~50% 차지)

## 2. 목표 지표 (생성 시 현실적 값으로 세팅)
- D1 리텐션 ~45%, D7 ~20%, D30 ~8%
- 결제 전환율 ~3~5%
- ARPU: 국가/채널별 차등 (KR/US > JP > 기타)
- 채널별 CPI 차등 (Organic=0, Facebook/Google 높음, Applovin/TikTok 낮음)

## 3. 테이블 스키마

### 3-1. `users` — 유저 마스터
| 컬럼 | 타입 | 설명 |

| user_id | STRING | PK |

| platform | STRING | iOS/Android |
| acquisition_channel | STRING | Organic/Google/Facebook/Applovin/TikTok |
| campaign_id | STRING | 캠페인 ID (Organic은 NULL) |

### 3-3. `events` — 행동 이벤트 로그 (핵심 Raw 데이터)
| 컬럼 | 타입 | 설명 |
|---|---|---|
| event_id | STRING | PK |

| session_id | STRING | FK |

**event_name 종류**: `tutorial_complete`, `stage_clear`, `level_up`, `gacha_pull`, `item_equip`, `item_upgrade`, `quest_complete`, `ad_watch_reward`, `shop_view`, `currency_earn`, `currency_spend`, `churn_flag`(내부 계산용)

## 4. 규모 (수정: 대용량 클라우드 처리 경험 어필용으로 확대)
- 유저 500,000명, 설치일 분산 365일간
- `events` 테이블 예상 로우 수: 수억 건 (유저당 활동일 x 일별 이벤트 수 기준)
- 로컬 생성 시 메모리에 한 번에 올리지 않고 **일자(day) 단위 청크로 생성 → Parquet 파일로 분할 저장** (BigQuery의 파티션 개념과 동일한 구조)

## 5. BigQuery 테이블 설계 (대용량 처리 반영)

| 테이블 | 파티션 키 | 클러스터링 키 | 비고 |
|---|---|---|---|
| `events` | `DATE(event_timestamp)` (일별 파티션) | `user_id`, `event_name` | 가장 큰 테이블, 파티션 프루닝 필수 |
| `sessions` | `DATE(session_start)` | `user_id` | |
| `purchases` | `DATE(purchase_timestamp)` | `user_id` | 매출 집계용 |
| `users` | 파티션 없음 (설치일 기준 클러스터링만) | `country`, `acquisition_channel` | 마스터 테이블, 상대적으로 작음 |
| `marketing_spend` | `date` | `channel` | |
| `ab_test_assignment` | 파티션 없음 | `experiment_name` | 상대적으로 작음 |

**적재 방식**: 로컬에서 일자별 Parquet 생성 → GCS 업로드 (또는 직접 `bq load`) → BigQuery 네이티브 파티션 테이블로 적재. 이 과정 자체가 "대용량 데이터를 클라우드 환경에서 다뤄본 경험"의 실증 자료가 됨.

## 6. 실제 GCP BigQuery 연동 안내
지원자(사용자)가 아래를 준비해야 실제 클라우드 연동이 가능:
1. GCP 프로젝트 생성 (콘솔: console.cloud.google.com)
2. BigQuery API 활성화
3. 결제 계정 연결 (무료 티어 내 사용 시 과금 없음: 월 1TB 쿼리 무료, 저장 10GB 무료)
4. claude.ai 커넥터 설정에서 BigQuery 커넥터 인증(OAuth) — 이 세션에서는 인증 플로우를 대신 실행할 수 없어 사용자가 직접 진행 필요

연동 전까지는 로컬에서 Parquet로 생성해두고, 연동 완료 후 그대로 BigQuery에 적재하는 순서로 진행.
